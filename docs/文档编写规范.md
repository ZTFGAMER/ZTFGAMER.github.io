# 功能实现类文档编写规范

本规范专门针对功能实现类文档，基于现有优秀文档实践总结，旨在保持文档风格的一致性和可维护性。

---

## 📋 文档命名

`{功能名称}.md`

示例：`变身礼包功能.md`、`竞技场机器人匹配系统.md`

---

## 📝 文档结构规范

### 整体结构
功能概述 → 核心功能 → 数据模型(可选) → 配置表(可选) → 系统集成 → 业务流程图(可选) → 总结

### 📋 功能概述
- 简短概括功能的用途和价值（1-3段）
- 说明功能的触发条件或使用场景
- 可以概述功能与哪些系统集成

### 🎯 核心功能
每个功能点包含以下子章节：

**功能说明**：
- 用无序列表列出核心要点
- 说明关键机制和规则

**实现说明**：
- 格式：`在 \`service/xxx.go\` 中的 \`MethodName\` 方法实现`
- 说明具体的实现文件和方法位置

**业务逻辑** / **业务规则**：
- 使用有序列表描述执行步骤
- 或使用表格/列表说明业务规则

**API接口**（如适用）：
- 接口路径和HTTP方法
- 请求参数：列出参数名和说明
- 响应数据：只列出核心业务字段（3-5个），格式：`- \`fieldName\`: 字段说明`

### 🗄️ 数据模型（可选）
使用表格展示数据模型：

| 字段 | 类型 | 说明 |
|-----|------|------|
| UserId | string | 用户ID |
| ... | ... | ... |

**相关文件**：标注模型定义的文件路径

### 🔧 配置表（可选）
使用表格展示配置表字段：

| 字段 | 类型 | 说明 |
|-----|------|------|
| ID | int | 配置ID |
| ... | ... | ... |

**相关文件**：标注配置加载的文件路径

### 🔄 系统集成
按子系统分组说明集成方式。

**每个子系统包含**：
- 方法名和位置：`方法: \`ClassName.MethodName\` 在 \`path/file.go\``
- 功能说明：简短描述方法用途
- 数据依赖（如适用）：说明依赖的字段或状态

**示例结构**：
```markdown
### 商店系统集成

**获取商店数据**:
- 方法: \`Shop.GetUserShopInfo\` 在 \`service/shop.go\`
- 功能: 获取所有商店数据，包括变身礼包

**数据依赖**:
- \`UserChapter.HeroMap\`: 存储变身次数
```

### 📊 业务流程图（可选）
使用ASCII流程图，用箭头和分步骤说明数据流转。

**基本格式**：
```
客户端调用 POST /api/endpoint
    ↓
Controller 验证参数
    ↓
Service 处理业务逻辑
    ↓
检查条件？
    ├─ 是 → 执行操作A → 返回响应
    └─ 否 → 执行操作B
            ↓
        更新数据
            ↓
        返回响应
```

**流程符号**：
- `↓`：顺序执行
- `├─` / `└─`：条件分支
- 简短的操作描述（如：客户端→DTO→Controller→Service→DAO→DynamoDB）

### 📝 总结
- **必须包含**，1-2句话概括核心内容、改进和成果
- 避免罗列要点，要有概括性
- 示例：`变身礼包功能通过变身次数记录、自动触发礼包生成、免费和付费任务分离、倒计时和过滤展示等机制，实现了完整的限时付费礼包系统，与商店、任务、章节、奖励、事务等多个系统深度集成，确保数据一致性和业务流程的完整性。`

## 🎨 格式规范

### 标题层级
- 一级标题：文档标题（1个）
- 二级标题：主要章节
- 三级标题：子功能或子系统
- 四级标题：谨慎使用，仅在确实需要更细分时使用
- **不跳级使用**

### 列表使用
- **无序列表**：功能列表、要点说明
- **有序列表**：流程步骤、操作顺序

### 表格使用
适用场景：
- 对比数据、配置参数、统计信息
- 业务规则对比（状态、条件、结果）
- 计算规则说明（类型、公式、说明）
- 任务类型/状态处理逻辑
- 数据模型字段定义（字段、类型、说明）
- 配置表字段定义

### Emoji图标
在二级标题前使用，保持文档视觉一致性：

| Emoji | 用途 |
|-------|------|
| 📋 | 功能概述/列表 |
| 🎯 | 核心功能/重点 |
| 🗄️ | 数据模型/存储 |
| 🔧 | 配置表/工具 |
| 🔄 | 系统集成/流程 |
| 📊 | 业务流程图/图表 |
| 📝 | 总结/文档 |

### 强调和标记
- **粗体**：重要概念、关键词
- `行内代码`：文件名、函数名、变量名、配置键
- 引用块：重要说明、注意事项

## 🚫 禁止内容

### ❌ 操作指南类章节
**禁止**：代码示例 / 使用示例 / 最佳实践 / 性能优化建议 / 使用说明 / 开发规范 / 配置方法

**原因**：功能实现类文档聚焦"是什么"和"怎么实现的"，而非"怎么用"

### ❌ 元信息和收益章节
**禁止**：相关文档 / 相关提交 / 核心收益 / 监控指标 / 性能测试 / 测试结果

**原因**：Git信息通过Git查看，收益整合到总结，测试数据易过时

### ❌ 代码块
**禁止**：代码片段、完整JSON/YAML

**例外**：可以展示配置示例、API请求/响应格式（简化版）

### ❌ 响应数据的错误写法
**禁止**：
- 展示完整JSON结构
- 使用表格列出响应字段
- 展开嵌套对象的内部结构
- 包含通用包装字段（code/data/message）

**正确写法**：
- 只列出核心业务字段（3-5个）
- 用列表形式：`- \`fieldName\`: 字段说明`
- 示例：`- \`heroCountMap\`: 变身次数映射`

### ❌ 代码位置链接
**禁止**：带行号链接 `[file.go:19-131](path#L19-L131)`

**正确**：只提及文件名或函数名

### ❌ 文档链接引用
**禁止**：相对路径引用、目录引用、章节锚点

**原因**：文档应自成体系，不依赖外部文档

### ❌ 文档元信息
**禁止**：在文档末尾添加版本、作者、更新日期等元信息

**原因**：通过Git管理即可

## ✅ 推荐内容

### 分点说明
按职责、位置、场景、规范等维度分点列出，提升可读性

### 对比表格
用表格对比不同状态、类型、条件的处理方式，清晰直观

### 业务规则表格示例
| 状态 | 条件 | 是否展示 |
|------|------|---------|
| 未购买已过期 | PurchaseCount == 0 && 当前时间 > DisplayExpireTime | ❌ 不展示 |
| 未购买未过期 | PurchaseCount == 0 && 当前时间 <= DisplayExpireTime | ✅ 展示，显示倒计时 |

### 计算规则表格示例
| 周期类型 | 计算公式 | 说明 |
|---------|---------|------|
| 1（天） | lastTriggerTime + displayCycleValue × 24小时 | 按天计算 |
| 2（周） | lastTriggerTime + displayCycleValue × 7天 | 按周计算 |

## ✅ 检查清单

编写完成后，检查以下项目：

- [ ] 文档命名符合规范
- [ ] 标题层级正确，未跳级
- [ ] 二级标题前使用了合适的emoji
- [ ] 文件名、函数名使用行内代码格式
- [ ] 表格用于合适的场景（规则对比、字段定义等）
- [ ] 无禁止章节（操作指南、元信息、收益、测试结果等）
- [ ] 无代码片段、完整JSON/YAML
- [ ] 无带行号的代码位置链接
- [ ] 响应数据使用列表格式，只列出核心字段
- [ ] 包含总结章节，1-2句话概括
- [ ] 总结有概括性，不是简单罗列要点
