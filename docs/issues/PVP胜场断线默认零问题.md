# PVP 胜场断线默认零问题

## 📋 问题概述

PVP 对战结束时，房主负责汇总所有玩家的胜场数并广播最终排名。当某个客户端断线或未在规定时间内上报胜场时，房主的兜底逻辑会强制广播当前已收集到的排名，未上报玩家的胜场数默认填为 `0`。

这导致排名结果无法区分以下两种情况，对玩家公平性造成影响：

- 玩家确实 **0 胜**（正常上报）
- 玩家**断线/超时未上报**（强制以 0 补位）

---

## 🎯 核心问题

### buildRankings 无法区分 0 胜与未上报

**问题说明**：

- `playerWins` 使用 `Map<number, number>` 存储玩家胜场，键为玩家 index，值为胜场数
- 客户端正常上报时通过 `wins_report` 消息写入 `playerWins`
- 断线玩家从不发送 `wins_report`，对应 index 在 Map 中无条目
- `buildRankings` 通过 `this.playerWins.get(p.index) ?? 0` 将未上报玩家胜场填为 `0`
- `0 胜上报` 与 `断线未上报` 在排名中完全相同

**实现说明**：

- 在 `PvpRoom.ts` 中的 `buildRankings` 方法实现

### reportWins 5 秒兜底直接广播默认值

**问题说明**：

- 房主上报胜场后，启动 5 秒超时定时器
- 超时后若 `gameOverBroadcast` 仍为 false，强制调用 `buildRankings` 并广播
- 此时未上报的断线玩家胜场均显示为 `0`，与真实的 `0 胜` 无法区分
- 玩家在结算界面看到断线对手显示 `0 胜`，可能误以为对方真的未赢过

**实现说明**：

- 在 `PvpRoom.ts` 中的 `reportWins` 方法和 5 秒 `setTimeout` 回调实现

---

## 🔄 系统影响

| 场景 | 当前行为 | 期望行为 |
|------|---------|---------|
| 玩家正常上报 0 胜 | 显示 0 | 显示 0 ✅ |
| 玩家断线未上报 | 显示 0 | 显示"未知"或特殊标记 |
| 玩家延迟上报（5s 后到达） | 已广播，上报被忽略 | 结算已发出，无法修正 |

---

## 📊 业务流程

```
所有玩家战斗结束
    ↓
各客户端调用 reportWins(wins)
    ├─ 客户端 → 发送 wins_report 给房主
    └─ 房主 → 直接写入 playerWins
    ↓
hostTryBroadcastGameOver()
    ├─ 所有人已上报 → 立即广播排名
    └─ 有人未上报 → 等待
    ↓
5 秒超时
    ↓
强制广播
    ├─ 已上报玩家 → 真实胜场
    └─ 未上报玩家 → 0（无法区分断线与0胜）
```

---

## 🎯 修复方向

- 将 `buildRankings` 返回结构中的 `wins` 字段改为可选（`wins: number | null`），使用 `this.playerWins.get(p.index) ?? null` 标记未上报状态
- 客户端排名展示时，对 `wins === null` 的玩家显示特殊标记（如 `--` 或 `断线`），与真实 `0 胜` 区分
- `PvpMsgToClient` 中 `game_over` 消息的 rankings 类型同步更新为 `wins: number | null`

---

## 📝 总结

`buildRankings` 使用空值兜底 `0` 导致断线玩家与零胜玩家在排名中无法区分，修复需将 `wins` 字段允许为 `null` 并在结算界面做相应展示处理，确保排名结果反映真实情况。
